## Problem Statement

If you have $10K in stocks and want to buy coffee, you have two options: sell your stocks (lose your position, trigger taxes) or take a margin loan ($1M+ minimum at Goldman, 2-4% interest, margin call risk). There's no way for a normal person to spend against their portfolio without selling.

The pitch: **"Goldman Sachs for everyone at 0%."**

## What Makes This Cool

Every payment becomes a micro-structured-product. When you tap to spend $50, an AI agent instantly:
1. Identifies the portfolio slice backing the spend
2. Wraps it in an options collar (bounded risk)
3. Lends you USDC at 0% interest
4. Executes the payment
5. Mints a "Spend Note" receipt showing the full structure

The platform profits from the collar spread (the gap between what the collateral is worth and what it lends). Because the collar bounds the downside, the platform can never lose money on the loan. That's why it's 0%.

Traditional SBLOCs: $1M minimum, 2-4% interest, margin call risk, takes days.
This: no minimum, 0% interest, no liquidation ever, instant.

**Nobody has built this.** MYSO Finance does zero-liquidation loans but charges interest (crypto only). Collar Protocol does collar-based lending but only stablecoin pairs. Stelrix has an investment-backed credit card but no options hedging and not 0%. The exact combination is novel.

## Collar Engine (the math)

Deterministic, not real options trading. For the demo:

```
Input:  stock_price = $225 (live from Yahoo Finance)
        spend_amount = $50

Step 1: Calculate shares needed
        shares = spend_amount / stock_price = 0.222 shares

Step 2: Set collar band (configurable, demo defaults)
        floor = stock_price * 0.95 = $213.75 (5% downside protection)
        cap   = stock_price * 1.15 = $258.75 (15% upside cap)

Step 3: Apply haircut (platform's profit margin)
        haircut = 0.05 (5% — configurable)
        collateral_needed = spend_amount / (1 - haircut) = $52.63
        shares = collateral_needed / stock_price = 0.234 shares
        advance = $50 (what user gets)
        platform_spread = collateral_needed - advance = $2.63

Step 4: Risk check (guarantees floor covers the advance)
        min_collateral_value = shares * floor = 0.234 * $213.75 = $50.02
        if advance > min_collateral_value: reject (too risky)
        else: approve
        (The haircut formula guarantees this passes:
         shares * floor = (spend / (1-h)) * (1-f) where f=floor%
         = ($50 / 0.95) * 0.95 = $50.00. With haircut=5% and
         floor=5%, they exactly cancel. Use haircut > floor% for
         safety margin. Demo default: haircut=5%, floor=4.5%,
         so min_collateral = $50.25 > $50. Always passes.)

Step 5: Convert to HTS integer amounts
        shares_hts = floor(shares * 1e6) = 234000  (MOCK-TSLA decimal 6)
        advance_hts = floor(advance * 1e6) = 50000000  (USDC-TEST decimal 6)

Output: "Collaring 0.234 TSLA shares ($215.06-$258.75)"
        "Lending: $50 USDC at 0%"
        "Platform spread: $2.63 (locked in at collar time)"
        "Max loss to platform: $0 (floor covers the advance)"

How the platform profits: The platform locks 0.234 shares ($52.63) but
only lends $50 USDC. The $2.63 difference is held in the escrow as
extra collateral. On repayment: user pays back $50 USDC, gets all
shares back, platform keeps the $2.63 buffer (transferred as USDC
from the escrow). On expiry/liquidation: agent sells 0.234 shares,
keeps $50 (covering the advance) + any remaining value, returns
excess to user. At floor price: collateral = $50.25, advance = $50,
platform still breaks even.
```

The floor protects the platform. The cap is what the user gives up. For small spends relative to portfolio, the cap barely matters.

## Token Architecture on Hedera

All via HTS (Hedera Token Service), no Solidity:

| Token | Type | Purpose |
|-------|------|---------|
| `MOCK-TSLA` | Fungible (HTS) | Mock tokenized Tesla equity, decimal 6 |
| `MOCK-AAPL` | Fungible (HTS) | Mock tokenized Apple equity, decimal 6 (portfolio display only, spend flow v1 is TSLA-only) |
| `USDC-TEST` | Fungible (HTS) | Mock USDC for lending/payments |
| `SPEND-NOTE` | NFT (HTS) | Receipt for each spend, metadata contains collar details |

**Escrow (simplified for hackathon):** The agent's server wallet acts as the escrow. No 2-of-2 threshold keys needed for the demo. Flow:
1. User's MOCK-TSLA shares transfer to agent's wallet (the lock)
2. Agent's wallet sends USDC-TEST to user (the advance)
3. On repayment: user sends USDC back, agent returns shares
4. On expiry (30-day TTL, not exercised during demo): agent keeps shares, sells for USDC

This is trust-based (user trusts the agent), which is fine for a hackathon. In production, this would use a dedicated escrow account with threshold keys or a smart contract. For the demo, the agent wallet IS the escrow.

**Why not 2-of-2 threshold keys:** Dynamic's embedded wallets manage keys internally. Extracting the user's raw key to construct a Hedera threshold key adds significant complexity and may not be supported. The simplified model (agent-as-escrow) demonstrates the same flow without the key management rabbit hole.

Agent wallet HBAR funding: pre-fund with 100 testnet HBAR at setup (covers ~2000 HTS transactions at ~0.05 HBAR each).

**Token association:** Before any HTS transfer, the receiving account must be associated with the token. The app auto-associates tokens during onboarding (user's embedded wallet associates with MOCK-TSLA, USDC-TEST, SPEND-NOTE on first login).

**Spend Note NFT metadata:**

HTS NFT metadata field is limited to 100 bytes on-chain. Full metadata stored on IPFS, CID stored in the HTS metadata field.

On-chain (HTS metadata, <=100 bytes):
```
ipfs://QmXxx...  (CID pointing to full JSON below)
```

Off-chain (IPFS JSON):
```json
{
  "name": "Spend Note #001",
  "asset": "MOCK-TSLA",
  "shares_collared": 234000,
  "shares_decimal": 6,
  "stock_price_at_spend": 225000000,
  "collar_floor": 215060000,
  "collar_cap": 258750000,
  "advance_usdc": 50000000,
  "platform_spread": 2630000,
  "created_at": "2026-06-14T15:30:00Z",
  "expires_at": "2026-07-14T15:30:00Z",
  "status": "active"
}
```

All amounts stored as integers (decimal 6) to avoid floating point issues. Status enum: `active` | `repaid` | `expired` | `liquidated`.

For the hackathon: use Pinata or nft.storage for IPFS pinning (free tier, instant).
